Defining Components

Components are POD-like structs:

struct Position {
float x{0.0f};
float y{0.0f};
};
struct Velocity {
float dx{0.0f};
float dy{0.0f};
};
struct Health {
int current{100};
int max{100};
};

Keep components small and trivially-copyable. Avoid pointers, virtual methods, and complex constructors.

Creating Entities

EntityManager em;
auto player = em.create();
em.add<Position>(player, {10.0f, 20.0f});
em.add<Velocity>(player, {1.0f, 0.0f});
em.add<Health>(player); // default-initialized
auto enemy = em.create();
em.add<Position>(enemy, {50.0f, 50.0f});
em.add<Health>(enemy, {50, 50});

Querying Entities

Iterate over entities with specific component combinations:

// Move all entities that have both Position and Velocity
em.query<Position, Velocity>().each([](auto& pos, const auto& vel) {
pos.x += vel.dx;
pos.y += vel.dy;
});
// Damage all entities with Health
em.query<Health>().each([](auto& h) {
h.current = std::max(0, h.current - 1);
});

Use par_each instead of each to run iterations in parallel:

em.query<Position, Velocity>().par_each([](auto& pos, const auto& vel) {
pos.x += vel.dx;
pos.y += vel.dy;
});

Defining Systems

A System is a class with an update method:

class MovementSystem {
public:
void update(EntityManager& em, float dt) {
em.query<Position, Velocity>().each([dt](auto& pos, const auto& vel) {
pos.x += vel.dx * dt;
pos.y += vel.dy * dt;
});
}
};
class DamageSystem {
public:
void update(EntityManager& em, float dt) {
em.query<Health>().each([](auto& h) {
if (h.current <= 0) {
// mark for removal in next tick
}
});
}
};

Scheduling Systems

The SystemScheduler runs systems in dependency order with parallelism where allowed:

SystemScheduler sched;
sched.add<MovementSystem>();
sched.add<DamageSystem>().depends_on<MovementSystem>();
// Per-tick:
sched.tick(em, 0.05f); // dt = 50 ms (20 Hz)

Destroying Entities

em.destroy(enemy);
// The entity ID is recycled with a new version; old references become invalid.
if (!em.valid(enemy)) {
log("entity destroyed");
}

Performance Tips

  • Keep components POD-like for cache-friendly iteration
  • Use par_each for systems with no inter-entity dependencies
  • Profile with tests/benchmark/ECS_* before optimizing
  • Avoid creating/destroying entities mid-iteration — defer to tick boundary
  • Group related components in queries to minimize sparse-set overhead
