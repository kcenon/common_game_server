Every MMO character needs an inventory — bag slots for carrying items, equipment slots for wearing gear, stackable consumables, durability, enchantments, and a catalog of static item templates. common_game_server provides this through the Inventory and Equipment components plus the ItemTemplate static data class. This tutorial walks through building an item catalog, spawning a character with an inventory, managing stacks, equipping gear, and calculating equipped stat bonuses.

Compilable version: examples/07_inventory.cpp

Static Item Catalog

ItemTemplate holds the shared, immutable definition of an item:

struct ItemTemplate {
uint32_t id = 0;
std::string name;
ItemType type = ItemType::Miscellaneous;
ItemQuality quality = ItemQuality::Common;
uint32_t maxStackSize = 1; // 1 = non-stackable
int32_t maxDurability = kIndestructible;
EquipSlot equipSlot = EquipSlot::COUNT; // COUNT = not equippable
StatBonuses statBonuses;
uint32_t requiredLevel = 0;
uint32_t vendorPrice = 0;
bool IsStackable() const noexcept;
bool IsEquippable() const noexcept;
};

Template data is static — you load it once at server startup from YAML/JSON/DB and reuse the same ItemTemplate across every stack of that item. The inventory component stores only item IDs and counts, not the whole template.

Build a small catalog inline for tests:

Call Initialize() exactly once after creating an Inventory component — it sizes the slots vector to capacity. After that, use AddItem and friends to modify contents.

AddItem uses a two-pass algorithm:

  1. Stack pass: for stackable items, walks existing slots and tops off stacks of the same item ID up to maxStackSize.
  2. Place pass: fills empty slots with the remainder.

It returns the number of items actually added (may be less than addCount if the inventory is full):

const uint32_t added = inv.AddItem(potion, 12);
// added may be 12 (fully placed) or smaller (inventory full)

InventorySlot

Each slot is a plain struct:

struct InventorySlot {
uint32_t itemId = 0; // 0 = empty
uint32_t count = 0;
int32_t durability = kIndestructible;
int32_t maxDurability = kIndestructible;
std::vector<Enchant> enchants;
bool IsEmpty() const noexcept;
bool IsBroken() const noexcept; // durability == 0 && maxDurability > 0
bool ReduceDurability(int32_t amount) noexcept; // returns true on break
StatBonuses GetEnchantBonuses() const;
void Clear() noexcept;
};

Enchant is a small bonus applied to a single slot, optionally time-limited. Use InventorySlot::RemoveExpiredEnchants on a timer to prune expired enchants.

Equipment is a fixed-size array — one slot per EquipSlot enum value. Indexing by static_cast<size_t>(EquipSlot::X). Equip returns the previously equipped item (possibly empty) so you can swap in one call:

Equipment& eq = equipment.Get(player);
InventorySlot sword;
sword.itemId = 2001;
sword.durability = 100;
sword.maxDurability = 100;
InventorySlot previous = eq.Equip(EquipSlot::MainHand, sword);
if (!previous.IsEmpty()) {
inv.AddItem(itemTemplate[previous.itemId], previous.count);
}

Stat Bonuses from Equipment

Equipment::CalculateStatBonuses(templates) walks every equipped slot, looks up the template, and accumulates the stat bonuses. Broken items (durability 0) contribute nothing — this is how the framework enforces that players must repair gear.

std::vector<ItemTemplate> allTemplates = { /* loaded at startup */ };
auto bonuses = equipment.Get(player).CalculateStatBonuses(allTemplates);
// bonuses.armor, bonuses.minDamage, bonuses.maxDamage,
// bonuses.attackSpeed, bonuses.attributes[...]

Enchant bonuses on a single slot are returned by InventorySlot::GetEnchantBonuses() and added to the item-level StatBonuses by CalculateStatBonuses.

Advanced: Splitting Stacks and Moving Slots

SplitStack(srcSlot, count) moves count items from the source slot into a new empty slot. Returns the new slot index or nullopt if either the source can't spare the items or no empty slot exists:

auto newSlot = inv.SplitStack(potionSlot, 5);
if (!newSlot) {
// inventory full or source is empty / too small
}

MoveItem(fromSlot, toSlot) swaps two slots or merges same-item stacks. It returns false if either index is out of range or equal. Use this for drag-and-drop UI.

Debugging Tips

  1. Print FreeSlots() after every mutation. Inventory bugs usually manifest as "I added an item but FreeSlots didn't change" — a sign that AddItem landed on an existing stack instead of a new slot, which may or may not be what you expected.
  2. Log item IDs, not pointers. InventorySlot is value-moved between Equipment and Inventory. Pointers go stale immediately.
  3. Assert template.equipSlot != COUNT before equipping. Equipping a non-equippable item (e.g., a potion) succeeds but creates a weird slot state that breaks CalculateStatBonuses.
  4. Unit test AddItem with a full inventory. The return value is the number actually added — easy to miss when addCount == 1 and you assume success.
  5. Breakpoint on Clear(). Slot clears are the most common "where did my item go?" source.

Common Pitfalls

  1. Forgetting Initialize(). A fresh Inventory has slots.empty(). AddItem handles this by calling Initialize internally, but FindItem / CountItem do not — they return 0 / nullopt on an un-initialized inventory.
  2. Stacking a non-stackable item. IsStackable() returns true only when maxStackSize > 1. Calling AddItem for a weapon with count = 5 creates 5 separate slots, one per weapon instance.
  3. Mismatched durability between template and slot. A newly added item inherits its template's maxDurability. If you later raise the template's maxDurability, existing slots keep the old value. Either repair to the new max or live with the discrepancy.
  4. Discarding the Equip return value. The return is the previously equipped item. Dropping it loses the item forever. Always hand it back to the inventory or explicitly destroy it.
  5. Broken items still visible. IsBroken() == true items display in the UI and are selectable — but CalculateStatBonuses ignores them. Warn the player with a "repair your gear" toast.
  6. Currency overflow. Inventory::currency is a signed int64_t. Subtracting below zero silently underflows. Clamp before subtracting: if (inv.currency >= price) inv.currency -= price;.
  7. Serialization of std::any enchants. Enchant is a plain struct, but enchantment effects are game-specific. Define an explicit (de)serializer if you plan to persist inventories.

Performance Notes

  • AddItem is O(slots) — at most two linear scans of the bag. For 40 slots that's 80 comparisons, well under a microsecond on any modern CPU.
  • FindItem / CountItem are also O(slots). If you need per-item-id O(1) lookup for crafting or quests, build a side-table keyed on itemId.
  • Equipment is fixed-size (kEquipSlotCount = 17), so all Equipment operations are effectively O(1).
  • CalculateStatBonuses allocates nothing. It accumulates into a stack-local StatBonuses.
  • SplitStack is O(slots) — needs an empty slot scan. For densely packed inventories this can be the most expensive inventory operation.
  • Enchant pruning is amortized O(enchants) via std::erase_if.
