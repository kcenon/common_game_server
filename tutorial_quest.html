Quests are the content engine of most MMOs — they give players direction, reward progression, and hook the combat / exploration / crafting systems together. common_game_server provides a data-driven quest system with templates, per-player quest logs, a 5-state quest state machine, pluggable objective types, and reward distribution hooks. This tutorial walks through every piece end-to-end.

Compilable version: examples/08_quest.cpp

The Quest State Machine

prerequisites is a list of quest IDs the player must have completed. chainNext unlocks the next quest in a chain on turn-in. flags uses a bitmask:

Flag Meaning
Repeatable Can be accepted again after turn-in
Daily Resets daily
Weekly Resets weekly
Shareable Party members can share progress
AutoAccept Auto-accepted when prerequisites met
Timed Has a time limit (use timeLimitSeconds)

Combine with bitwise OR and test with HasQuestFlag:

wolfQuest.flags = QuestFlags::Daily | QuestFlags::Shareable;
if (HasQuestFlag(wolfQuest.flags, QuestFlags::Daily)) {
// schedule reset at next server day-rollover
}

QuestLog Component

Every player carries a QuestLog:

struct QuestLog {

Accept performs three checks before adding the quest:

  1. The log is not already at maxActiveQuests capacity.
  2. The quest is not already in activeQuests.
  3. CanAccept(tmpl) — prerequisites + not-completed (unless Repeatable) + level requirement.

On success, a fresh QuestEntry is appended with the objectives copied from the template.

Tracking Progress

Progress can be recorded two ways:

Option A: Direct UpdateObjective call (simple cases):

auto* entry = log.GetQuest(questId);
entry->UpdateObjective(ObjectiveType::Kill, /*targetId*/ 3001, /*amount*/ 1);

UpdateObjective walks the entry's objectives, finds the ones whose (type, targetId) match, advances current by amount, and returns true if any objective was updated. When every objective becomes complete, the entry's state flips to ObjectivesComplete automatically.

Option B: Queue a QuestEvent (systems integration):

struct QuestEvent {
Entity player;
QuestEventType type = QuestEventType::Kill;
uint32_t targetId = 0;
int32_t count = 1;
bool processed = false;
};

Other systems (CombatSystem on creature death, InventorySystem on item pickup, WorldSystem on zone entry) emit QuestEvent components which QuestSystem routes to the correct player's log each tick. This is the decoupled production path — your combat code does not need to know about quests.

Turn-In and Rewards

When objectives are complete the player returns to the quest giver and the game calls:

if (log.TurnIn(wolfQuest.id)) {
// Distribute rewards manually.
// QuestLog::TurnIn only flips state and records completion —
// actually granting XP / currency / items is your responsibility.
stats.Get(player).experience += wolfQuest.rewards.experience;
inventory.Get(player).currency += wolfQuest.rewards.currency;
for (auto [itemId, count] : wolfQuest.rewards.items) {
inventory.Get(player).AddItem(templates[itemId], count);
}
}

TurnIn returns false if the quest is not in state ObjectivesComplete. It sets state to TurnedIn and inserts the quest ID into completedQuestIds, so future CanAccept checks honor the "already done" rule.

Chain Quests

Quest chains are implemented by QuestTemplate::chainNext. When a quest turns in, your game code looks up chainNext and offers the next quest to the player:

if (log.TurnIn(q.id) && q.chainNext.has_value()) {
const auto& nextTemplate = templates[*q.chainNext];
log.Accept(nextTemplate); // auto-accept the next link
}

Alternatively, set QuestFlags::AutoAccept on the chain link so the system offers it automatically.

Debugging Tips

  1. Log state transitions. Any time a quest moves between states (Accept, all-objectives-complete, TurnIn, Abandon), emit a CGS_LOG_INFO(LogCategory::Core, ...) with the player ID and quest ID. Saves hours of "why is this quest stuck" debugging.
  2. Print objective.current / required when a player complains they can't turn in a quest. The issue is almost always one objective short.
  3. Verify (type, targetId) matching. Objective progress updates match on the pair exactly. An off-by-one in the creature entry ID silently no-ops — the objective stays at 0/5 forever.
  4. Check completedQuestIds for phantom completions. A quest you abandoned mid-progress should not be in the completed set. If it is, you called TurnIn somewhere you shouldn't have.
  5. Trace QuestEvent processing. If you use Option B (event queue), the most common bug is QuestEvent components that never reach the player's log. Add a counter in QuestSystem for events-processed and compare against events-queued.

Common Pitfalls

  1. Mutating QuestEntry::objectives directly. Always use UpdateObjective or the QuestObjective::AddProgress helper so the completion state flags update correctly.
  2. Forgetting to grant rewards. TurnIn does NOT automatically distribute XP, currency, or items. Your game code must do that step. An easy safety net is a wrapper function TurnInAndReward(log, stats, inv, q).
  3. Re-accepting a non-repeatable quest. CanAccept checks completedQuestIds — but only if the template does NOT have QuestFlags::Repeatable. Mis-setting the flag can let players farm the same quest.
  4. maxActiveQuests overflow. Default is 25. Going over makes Accept return false silently. UI should warn the player and refuse to show more accept dialogs.
  5. Daily/Weekly resets. QuestFlags::Daily is a tag — the framework does not implement the reset job. You must schedule a daily task that iterates every player's completedQuestIds and removes daily quest IDs.
  6. Objective types with invalid targetId = 0. Some tools export quests with missing target IDs. UpdateObjective matches on targetId == 0 and may update unintended objectives. Validate templates at load time.
  7. Timed quests with timeLimitSeconds == 0. QuestFlags::Timed but zero limit means "no limit" — the timeout never fires. Reject such templates at load time or strip the flag.

Performance Notes

  • Accept is O(activeQuests + prerequisites). With 25 active quests and single-digit prerequisites, this is a handful of comparisons per accept.
  • UpdateObjective is O(activeQuests × objectives) in the pathological case, but in practice most events match 0–1 objectives and early exit.
  • completedQuestIds uses unordered_set — O(1) membership check.
  • QuestEvent processing is batched — all events for a tick are drained in one QuestSystem::Execute pass.
  • Memory per active quest is ~256 bytes (the objectives vector dominates). 25 active quests × 256 B = 6 KB per player, trivial.
  • Quest templates are shared static data. Allocate once at boot and keep a std::vector<QuestTemplate> or std::unordered_map<uint32_t, QuestTemplate> for lookup.
